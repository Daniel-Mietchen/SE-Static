How does the kernel recover should a device driver gets stuck in the interrupt?

up vote 3 down vote favorite
1

In case, if a device driver gets stuck in handling the interrupt, what practical provisions exists in Linux kernel to recover within time?

EDIT1: The "stuck" word is relative. However, as far as interrupts are concerned, they should be typically processed within few milliseconds at best, so any large enough time even if the process is not completely hang is still "stuck" in the context.

EDIT2: Please note, I am not asking which other external management to achieve this. The question is what 'smartness' does the kernel has to deal with it directly.

It largely depends on the Linux Kernel that is compiled for your target processor and what that processor offers.

There are Linux Kernels, especially in the more Real-Time OS distro's that use available timer structures to maintain an overview of the success of operation of all things that go on. It will then allow for forcing things to complete or terminate after a certain time-out.

But, if something gets stuck inside a hardware interrupt on a less complex processor, it may well be there is no way to recover. A common way in "home-flavour" Micro Kernels on smaller devices is to enable a Watchdog Timer Interrupt and add some checking in that. Since Watchdog interrupts are usually very high on the table of priority. On many devices those checks can then turn off any blocking interrupts in hardware and redirect operation elsewhere, based on internal state memory to operate as glitch-free as possible. This is either very dirty or an awesome solution, depending on how you look at it.

In most cases the interrupt should just be a notifier or a simple data-copy that then allows execution of special code in non-interrupt space, which makes terminating badly behaving driver code much easier to do without causing problems elsewhere.

The answer depends on the type of interrupt routine that is "stuck" and the hardware architecture. There is usually a hierarchy of interrupt priorities, and a low- or mid-priority interrupt handler will eventually get pre-empted by a higher priority interrupt, which could be either a hardware event or a kernel-controlled timer. At the highest priority one could expect to find a watchdog timer specifically for recovering from unexpected hangups.

However, I am not sure how (or even if) the Linux kernel would actually detect that a specific (lower priority) interrupt routine was stuck, or clean up from such a situation. A higher priority interrupt could keep the system partially responsive, or at least allow the scheduler to run at some point, at which time a problem might be noticed. In some systems a hardware watchdog will reset the entire processor, which would certainly "unstick" things.
