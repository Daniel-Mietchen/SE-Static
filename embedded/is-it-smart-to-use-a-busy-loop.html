Is it smart to use a busy-loop?

up vote 12 down vote favorite
1

Occasionally in code for embedded systems, I will see something like this:

loop:
add a, 1
cmp a, b
jl loop

As a way of stalling the execution of the code for a certain amount of time.

Is this good practice? I think this might be okay because...

  • With embedded systems programming, you usually know all about the processor that you are working on. Therefore, you can find out how long it takes for the instructions being used to execute (in clock cycles), find out the speed of the processor's clock, and, with a few calculations, be able to find out how many times to iterate to wait for x seconds.

However, would it be preferred to use a timer interrupt or some sort of system clock over this method? If so, why?

Sometimes.

The big upside to busy-waiting is you get fine-grained control over the delay. If you need to wait a few hundred nanoseconds for a line to stabilize, this lets you do it.

The big downside is that busy-waiting uses power. If you're waiting for more than a few milliseconds, you're probably better off setting up a timer interrupt or equivalent, and putting the CPU in a low-power sleep state.

This is, of course, assuming a single-threaded environment. If you're multi-threading, the fine-grained control goes away.

Short Answer
These kinds of delay loops should only be used as a last resort and on the simplest of systems. I define "simplest" as single-threaded with only one core, interrupts disabled, no sleep or low-power states available, and no timer or real-time clock (RTC) interrupts available.

Drawbacks
If you have multiple threads or multiple cores, then you must have some kind of scheduler that could make a context switch away from the wait loop, throwing off the timing.

Even if you're working on one core with one thread, interrupts can also throw off the timing if an interrupt needs to be serviced while the loop is running.

As other answers pointed out, spinning on a loop also consumes extra power, so if entering a sleep state is an option, that would be a better choice.

Since they can be unreliable, they can also lead to bugs that can be difficult to identify and debug. If your timing loop is just long enough to work most of the time but sometimes fails (because of the reasons previously outlined), that failure can be difficult to isolate and reproduce.

Advantages
One upside to the delay loop is that it is simple. It can get the job done if you have no timer or RTC interrupts available.

Additionally, writing the code is pretty simple. It can be a decent stand-in for more reliable methods while bringing up a system early in development, with the intention of replacing it with a better solution later on.

Conclusion
Since there are multiple disadvantages to the loop but no significant advantages besides simplicity, they really shouldn't be used long-term unless there aren't any other options.

It's very typical to have ways to delay inside an embedded system, however in most scenarios there are better ways to go about this including timers and other interrupts. To show both the pro's and con's of this in terms of when timing matters like serial communication.

There are certain devices that depending on how they are designed will have trouble getting a stream of data or may be expecting a certain delay between bytes sent. If you're planning on using them, you'll need to conform more to their styles and busy waiting is typically the most efficient way to make sure your timing is right.

In most cases it doesn't make sense to busy wait while receiving so you can actually run your system and process the data you are getting. In cases like this you let main loop run though and have an interrupt put bytes in a buffer to be processed later and once the data is received, you can process it. Also in this scenario you can set up a timer to set a flag on a timeout scenario where you need to resend you last message rather than just waiting checking if the data is valid at the end of a busy wait when you could be checking other things.

So you have to weigh the options and restrictions of your system and the drives you interact with. One last thing to note although you can just put a simple for loop in or while loop to tick down, if you really need to delay it's better to figure it out once and configure it to a function. Here is a simple example on an MSP430 chip.

// source clk 16MHz
unsigned int ClkFreq = 16;
void DelayMs(unsigned int millis)
{
   unsigned int count = 1000;
   while(millis--){
      while(count--){
         __delay_cycles(ClkFreq);
      }
      count = 1000;
   }
}

The __delay_cycles function is just a nop and could be used just where you need it, but wrapping this concept in a simple to call function makes it very explicit how long you are delaying without too much thought.

Delay loops are useful if the delays are very short with respect to the processor speed. Considering the ARM architecture, if delays are in microseconds, the overhead for setting timer interrupts and multiple interrupt sources may affect the loop timing. Hence for very small fractional delays, a delay loop is the best option.
