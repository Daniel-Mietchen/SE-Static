Does IAR embedded workbench's watchlist directly access registers?

up vote 7 down vote favorite

I'm working on an ADC that converts 3 different measurements. It is configured in a way so that the converter writes the result into a data register and an interrupt is invoked when it is finished.

After that the converter idles because he has a flag set that forbids him to write further values into the data register before it is read. Also it is not allowed to discard any value and the order in which it converts the values is determined, so I can predict which incoming value corresponds to which conversion.

The interrupt routine looks like this:

int resultOf1;
int resultOf2;
char resultOf3;

void interupt ()
{
    static int indexI = 0;

    switch ((indexI % 3))
    {
        case 0:
            resultOf1 = RegisterStructure.DATA;
            break;
        case 1:
            resultOf2 = RegisterStructure.DATA;
            break;
        case 2:
            resultOf3 = RegisterStructure.DATA;
            break;
        default:
            SomeErrorLogging();
            break;
    }

    indexI++;
}

The program works fine, but as soon I'm debugging the interrupt, it is desyncing and assigning strange values.

I came to the conclusion that this is probably because the IAR debugger is accessing the watchlisted DATA register which is treat by the system as reading it and just writing in the next conversion value as the previous had been read (by IAR, and not by my software) while my software still is waiting on a breakpoint.

Can anyone confirm my conclusion or tell me what else might be the error? If I'm right, is there another way I could watchlist DATA without IAR notifying the system about the reading access?

In the "Variables and Expressions" chapter of the C-SPY Debugging Guide, it says (page 89):

The Watch window allows you to monitor the values of C-SPY expressions and variables. The window is automatically updated when execution stops.

And in the section on the Watch window (page 102):

Every time execution in C-SPY stops, the values in the Watch window are recalculated. Values that have changed since the last stop are highlighted in red.

Since special function registers are memory mapped in the ARM architecture, the debugger must be reading that memory location to get the current value whenever a breakpoint is hit or you manually stop execution.

The feature where you can hover over a variable with the mouse is called "Tooltip Watch". The user guide doesn't say it, but the variable's value is only available when execution is stopped so it's reasonable to assume that it's using the same mechanism to access, i.e. a read from the memory location.
