How can I know the cause of a hard fault exception on an ARM Cortex-M3?

Hard fault is an exception that can be caused by various reasons. How can I know the real cause when it occurs?

The cause of a hard fault exception can be determined from the HardFault Status Register (0xE000ED2C). If the FORCED bit (0x40000000) is set, then you need to read the Configurable Fault Status Register (0xE000ED28) to determine the cause of the fault.

For example, if you have a divide by zero, CFSR will have the 0x02000000 bit set.

By the way, this answer also applies to a Cortex-M4 (and probably Cortex-M0).

If you want to know the code that triggered the hard fault, you need to determine the PC when the hard fault occurred. This can be accomplished with the following Ada code:

type Hardware_Context is record
   R0, R1, R2, R3   : Word;
   R12, LR, PC, PSR : Word;
end record;

procedure Hard_Fault_Handler_Asm;
pragma Machine_Attribute (Hard_Fault_Handler_Asm, "naked");
pragma Export (Asm, Hard_Fault_Handler_Asm, "__gnat_hard_fault_trap");
--  This assembly routine needs to save and restore registers without
--  interference. The "naked" machine attribute communicates this to GCC.

procedure Hard_Fault_Handler (SP : Address);
pragma Export (Asm, Hard_Fault_Handler, "__hard_fault_handler");

procedure Hard_Fault_Handler_Asm is
begin
   Asm ("tst lr, #4"                    & NL &
          "ite eq"                        & NL &
          "mrseq r0, msp"                 & NL &
          "mrsne r0, psp"                 & NL &
          "ldr r1, [r0, #24]"             & NL &
          "ldr r2, handler_addr"          & NL &
          "bx r2"                         & NL &
          "handler_addr: .word __hard_fault_handler" & NL,
        Volatile => True);
end Hard_Fault_Handler_Asm;

----------------------------
-- Hard_Fault_Handler --
----------------------------

procedure Hard_Fault_Handler (SP : Address) is
   Hw_Ctx : Hardware_Context with Address => SP, Volatile;
   HFSR : Interfaces.Unsigned_32 with Address => 16#E000_ED2C#, Volatile;
   CFSR : Interfaces.Unsigned_32 with Address => 16#E000_ED28#, Volatile;
begin
   Asm ("BKPT #01", Volatile => True);
   --  SAM4S.Trace.Put (0, "In Hard_Fault_Handler");
   --  SAM4S.Trace.Put (0, SAM4S.Trace.Value_Type (HFSR));
   --  SAM4S.Trace.Put (0, SAM4S.Trace.Value_Type (CFSR));

   if (HFSR and 16#4000_0000#) /= 0 then
      --  Forced hard fault.
      if (CFSR and 16#0200_0000#) /= 0 then
         --  Divide by zero.
         --  SAM4S.Trace.Put (0, "Divide by zero.");
         null;
      end if;
   end if;
end Hard_Fault_Handler;

__hard_fault_handler is in the exception vector (for the hard fault exception).

Hard_Fault_Handler forces a breakpoint when it is executed so that you may inspect the Hw_Ctx in a debugger.

Note that if you have a UNDEFINSTR usage fault, the PC won't help you. For that, you will likely want to capture and view the ITM trace information.

https://blog.feabhas.com/2013/02/developing-a-generic-hard-fault-handler-for-arm-cortex-m3cortex-m4/

http://www.freertos.org/Debugging-Hard-Faults-On-Cortex-M-Microcontrollers.html

http://infocenter.arm.com/help/index.jsp?topic=/com.arm.doc.dui0553a/BABBGBEC.html

If you can connect a debugger to the device (eg, gdb via JTAG), you can directly inspect the System Control Blocks's HardFault Status Register (SCB->SHCSR), which is ARM-specified. The locations and contents of these registers are vendor agnostic, eg present in both ST and NXP devices; you can read more about them on ARM's website, and Keil has a good app note (pdf) that is good background.

For example, to print the System Handler Control and State Register (SHCRS), Configurable Fault Status Register (CFSR), and HardFault Status Register (HFSR) with gdb:

(gdb) set mem inaccessible-by-default off
(gdb) x 0xE000ED24
(gdb) x 0xE000ED28
(gdb) x 0xE000ED2C

Patrick's answer gives an example of how to find where in your firmware the hardfault originated from (what program counter memory location). If you don't have a hardware debugger, his code could be extended to print out memory locations via a UART (serial) port. I find that a gdb backtrace sometimes works (citation needed), and if that fails it's usually possible to figure out what went wrong by inspecting more of the SCB registers or bisecting .
