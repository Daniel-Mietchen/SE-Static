Code composer studio Passing array

up vote 1 down vote favorite

I am writing a communication protocol library which uses 8 but CRC. I am using table based CRC calculation method. For testing with multiple CRC algorithms I am storing pointer to CRC table rather than full array. During run time I can assign different table address for testing code. 

//Function in communication protocol library
    char * modcrctab ;
    void Module_Init(const char * crctab )
    {
        modcrctab = crctab ;
    }
//Crctablefile
const char Crc8Table[256] = {/* values .... */}

Code composer 6.1 gives warning when I try to use function as

169-D argument of type "const uint8_t **" is incompatible with parameter of type "const uint8_t *"

Module_Init(&Crc8Table) ;

But program works as required. When I write following code

Module_Init(&Crc8Table[0]) ;

There is no warning but value assigned to modcrctab is not actual address of table.

Am I using proper syntax ?

crc8table is the name of your array. When used in a function call the name of this array will its self decay into a pointer. So having &crc8table is redundant. You are referencing something that is already a pointer, which means you are passing a double pointer. In this case, what you are doing would be incorrect. Either the compiler could ignore the extra reference operator and generate code based on what it thinks you are trying to do or it could throw an error. It looks like ccs chooses to throw an error which is good.

Other stuff:

&Crc8table[0] compiles because the array notation with the [] brackets is syntactic sugar for the * dereference operator. So Crc8table[X] effectively turns into *(Crc8table+X). By putting the & in front it turns into &(*(Crc8table+X)). Which should work according to operator precidence. The [] should be evaluated before the &.

Consider the following:

void main(void)
{
    char my_array[] = {"Hello World"};

    printf("0x%x\n", my_array;);
    printf("0x%x", &my_array[0];);
}

I compiled this using IAR EWARM 7.4 for an ARM cortex M4 Target, and here is the output.

  0x20001ff0
  0x20001ff0

So, therefore, accessing the pointer to the head of the array directly or by dereferencing and rereferencing the first element should yield the same result. It is just a redundant way of doing things. But I am not really sure why that didn't work out for you. It should have. Perhaps I am missing something.
