4-bit checksum for simple comms protocol?

up vote 4 down vote favorite
1

I have a simple serial comms protocol consisting of 4 byte messages:

  • Controller ID (0-255)
  • Command ID (0-15)
  • Data 1 (0-255)
  • Data 2 (0-255)

I have 4 spare bits available for some error detection code (CRC, Checksum, or otherwise).

I would like to know some pros and cons of different error detection techniques given the scenario of a 3.5byte data word and 4bit detection code.

My specific application is a multi-drop half duplex RS485 interface where collisions are possible which can lead to multi-byte errors. But I am asking this question in general terms. Assume this protocol could operate over any interface.

I agree with others on using a 4-bit CRC; if that seems too hard to code up you can modify Fletcher's checksum to generate a 4-bit checksum. The CRC is stronger mathematically, but we used Fletcher-16 on a distributed serial radio project to good effect. See the Fletcher article on why simple checksums are weak.

Philip Koopman maintains a table of "good" CRC polynomials. That suggests x^4 + x + 1 as a good polynomial to use for CRC-4. Put that polynomial (10011) into Kay Gorontzi's CRC code generator and it will write the C code for you!

(You might want to be a bit freer with your total message size. Adding packet length will make it a lot easier to extend your protocol in the future, and it's just easier to work with bytes than nibbles.)
