Is there a simple threading abstraction available for microcontrollers like the 8051?

up vote 1 down vote favorite
2

In my last question How can I emulate Virtual memory mode when working with microprocessors like 8051?, I learnt that there is no straightforward and productive way for creating virtual memory management in 8051-type microcontrollers. I needed this essentially to see if there are good ways to capture segmentation faults, if any.

Now the next in line might be at least more reasonable to ask for - is there a way to emulate threading in an elementary microcontroller? From what I have learnt, uCLinux doesn't quite provide that but are there any other systems that do?

My rationale for asking this is, while 8051-type microcontrollers don't provide protected mode hence no MMU, they do provide interrupts. Hence it should be feasible to create and elementary scheduler that can help allow simple threading.

Is this possible? Are there any known solutions?

It isn't difficult to set up a multithread system. Programming in a multithread system is the difficult part!

In concrete terms, a thread is a stack and the state of the registers that it uses (PC (program counter), SP (stack pointer), general-purpose registers, memory configuration registers, etc.).

To switch between threads, you save the registers of the current thread somewhere (either a dedicated memory area, or on the thread's stack), and you restore the registers of the target thread from where they were saved. This task has to be performed by the kernel (at least in part) on a system with memory protection — the kernel has to at least switch the memory configuration registers. On a system without memory protection, who does this is a moot point. The set of registers (and if applicable OS settings) associated with a thread are called the thread's context, hence the name context switch for switching between threads.

There are two regimes for when a switch between threads happens:

  • In cooperative scheduling, a thread runs until it decides to yield to another thread. It's easier to get programs right with cooperative scheduling because a thread can only be interrupted at times that the programmer decides — so for example you don't need to worry about interruptions while some data structure is inconsistent. On systems with extremely limited resources, cooperative scheduling has the advantage that fewer registers need to be saved, because a thread can decide to yield only when it has few registers with data that needs to be preserved.

    Even systems with cooperative scheduling allow interrupts to be raised most of the time, but generally an interrupt will only set some flags and perform some urgent operations before reenabling interrupts and resuming the current thread. The rest of the reaction to the interrupt only happens after the next yield.

  • In preemptive scheduling, a thread can be interrupted at any time, either by a timer interrupt or any other interrupt or during a system call. The part of the system that performs the context switch must save all the registers and other settings, and restore the context of the new thread.

    It's harder to write programs for systems with preemptive scheduling: they're prone to race conditions. On the other hand, these systems are more robust in that even if a thread is buggy and goes into an infinite loop, that doesn't prevent other threads from running.

A system can mix preemptive and cooperative scheduling, for example to have cooperative scheduling for hard real-time tasks and preemptive scheduling for the rest.

To write a thread abstraction, you need two pieces. One is the context switch mechanism; this can be done in a few lines of assembly. The other piece is the scheduler: the piece that decides which thread to do next. Schedulers can get very complex, but it's possible to write simple schedulers for a broad range of uses.

A typical simple scheduler maintains a list of runnable threads — threads are runnable if they're ready to execute something. The scheduler picks a runnable thread and executes it (if no thread is runnable, the scheduler sleeps until the next interrupt). Threads are not runnable if they're blocked on something, e.g. waiting for a flag raised by an interrupt (raising that flag would mark the thread as runnable) or waiting for a message from another thread (the message passing mechanism would mark the recipient thread as runnable).
