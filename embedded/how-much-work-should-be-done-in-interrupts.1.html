How much work should be done in interrupts?

up vote 22 down vote favorite
2

I have a project that I am working on where I using an interrupt to collect bytes of data over UART. The protocol is just typical ascii and always starts and ends a message with a return and a new line byte.

I have two designs right now for the interrupt one where the bytes are just put in a buffer when they are read and processed later and one where the interrupt processes the data and checks what state it is in to determine if the byte is data or not and updates its state.

What are the pro's and con's of each of these scenarios?

Simple answer
In most cases, the correct answer is "as little as possible".

Details
As little as possible should be done in an interrupt handler because while your CPU/microprocessor is running in an interrupt context, everything else is on hold. Even if you're working on a multithreaded system that can handle multiple simultaneous interrupts, it is probably still limited to only one interrupt per interrupt source at once. In your particular case, this means that you risk missing incoming packets over your UART while you're processing one packet.

Some microprocessors allow interrupt handlers to interrupt each other, so you could configure it to repeatedly interrupt your handlers as new packets come in. This has several drawbacks, however:

  1. You have to write your handlers to be reentrant, which can be a challenge for complex operations.

  2. There is likely a limit to how many interrupts handlers can run simultaneously, and it's probably much smaller than your normal function stack.

  3. Having this many asynchronous events layered makes things difficult to debug. Having the interrupt handler load data into a buffer for later processing is a much more simple approach.

If you have a simple enough system where dealing with these packets is the only thing it ever has to do, then you can probably get away with it. But then you're making your system more difficult to maintain and improve in the future. It will be harder to add future functionality if you design it in a way that intentionally hogs system resources.

The answer here depends on a number of factors, and it really is a matrix of tradeoffs. The perceived wisdom is that you should do as little as possible in an ISR, but there are many cases where that does not hold.

Some factors include:

  • Power source.
  • Number of interrupt sources
  • Minimum latency requirements
  • Non real-time processing requirements

Power Source

Consider a battery-powered, micro-controller based system, where time between recharging is a major marketing factor. In this case, you want your processor to sleep as much as possible. Interrupts will typically wake a processor from sleep (with some extra latency).

In this case, you want to do as much as possible in interrupts and your overall software design should be event-driven. Basically, your mainline should initialize the chip and its peripherals, and then go into a sleep loop waiting for interrupts.

If you have multiple interrupts, and are very careful, you can actually re-enable higher-priority interrupt sources in your ISR to allow nesting.

If you are mains powered, and not trying to meet Energy Star (or equivalent) requirements, then you can shunt all low-latency work to the background and keep your ISRs short. You can even use polling instead of interrupts where suitable.

Number of interrupt sources

If you only need to deal with a single interrupt source, then again you can do a lot in the ISR. This assumes anything which is running in the background can afford to be delayed while you handle the interrupts.

If you have multiple interrupt sources, most controllers will not "lose" an interrupt. Instead it will queue them and deliver the highest priority one as soon as you exit the ISR. However, you might lose interrupts if multiple triggers occur before you acknowledge the first one.

In the worst case, the added latency of waiting for running ISRs to complete can cascade so that you never return to the background. This type of scenario can, again, require you to enable interrupts during an ISR to allow high priority triggers to be serviced.

Latency Requirements

This is a double edged blade. If you have tight latency (i.e. hard real-time) requirements, then you need to keep your ISR's short so that as much of the processor is available as possible for dealing with interrupts.

On the other hand, if an input requires you to respond within a very short window, you may need to deal with that particular interrupt completely within the ISR.

Background Processing

Now consider an application where your interrupts provide data, but that it needs significant processing. This is where you want a short ISR that buffers the data and lets the background chew on it at will.

Here you want all your ISRs to be short.

Remember that time spent in interrupt routines is time when your background code is not running. The best way to visualise this is by considering the virtual clock speed for your background code.

Let's say your micro-controller has a 4MHz crystal with an internal divide-by-4. So you effectively have a 1MHz chip. If it spends 50% of the time in ISR's, the background code is only running on the equivalent of a 500KHz processor. A 25% interrupt load means you have a 750Khz processor, that's a 50% improvement in apparent processing power for the background code.

For your UART

Assuming there are no timing requirements, I would normally just drain the UART into a FIFO and let the background handle it. Most UART chips have a built-in FIFO of 2 or 4 characters, but this can quickly be overrun if you have other interrupts to deal with, so be sure to completely drain the UART on each interrupt.

I'd also use a FIFO for transmission, simply loading the first character into the UART and letting the TDRE interrupt continue to drain the FIFO.

This allows you to worry about implementing the actual protocol in the background, while leaving the interrupt domain free for other - possibly more important - tasks.

In general, you would want to do as little work as possible inside of an interrupt. An interrupt does just what it sounds like; it interrupts whatever was happening before.

In your case, you would probably want to go with scenario #1, this will allow the interrupt to finish quickly, allow the processor to get back to whatever it was doing before, and then process the buffer when it has time, especially if new data is going to be coming in quite frequently.

Now, if the data is not very long, then it probably doesn't matter too much, but it's better practice to keep any heavy lifting out of the interrupt and only do things in the interrupt that must be done in the interrupt.

Now, if you need something done immediately on interrupt, like some system critical code that is a bit heavier, that might be a time when you would do it all inside the interrupt. However, in that scenario, you'd probably want to be using an RTOS that guarantees your critical tasks are executed when you need them to be.

Hope this helps. I'm an embedded systems major in my senior year (starting masters soon), so some things may not be exactly true (feel free to correct anything if anyone knows more), but this is all from my experiences working with various systems.

As little work as possible should be done in an interrupt handler (aka. an ISR, "interrupt service handler").

Many/most contemporary operating systems make interrupt handlers non-preemptible, and abusing this privilege amounts to trying to circumvent the scheduler. If you are doing that purely to process data that is easily buffered, it is inappropriate.

Related to this fact, and of particular significance for your scenario, is that handling an interrupt may disable the handling of the other interrupts while the handler handles. This might easily amount to data being lost.

The difference in the advice would be determined by: How much else will your system eventually do?

Most small systems (some specialty processors or multi-core processors excluded) have only one layer of interrupt, so running one interrupt will lock out all others on those systems.

The point of interrupt routines is to make critical things go quick and efficient, and in many cases in a determinable time. If you then have an interrupt routine that runs for a milisecond, or even a tenth of it, your system will lock out other interrupts for that time, which already makes the delay statistically hard to determine.

Let alone when you go into multiple miliseconds or even tens of them, locking out others that may be more critical will make it (almost) human noticeable.

So if your application is just get UART data, calculate something, send data back when done, it's all not that relevant. If your current design is just the first phase of a system that will be using many more pieces of hardware with many more interrupt sources, it's always better to do "what needs to be done right now!" (copy the byte to a buffer to make sure you don't lose it) in the interrupt and offload the rest to a part of code that can be interrupted by others that have a "right now" situation.

I can think of one case where the processing in the interrupts should actually do more than just push the data to the buffers: Most data rejected.

Say, it's a module plugged into a bus with many modules. It receives lots and lots of messages which are not addressed to it. In this case the interrupt can peek into the received data and just reject the packet if it's "not yours", saving a lot of buffer space and quite a bit of "base level" processing time. And even that only if the protocol is simple enough that determining the address is trivial. If you need to decompress the frame first, that's too much time spent in the interrupt doing things it should not be doing.

Of course verifying the checksums and requesting a re-send in case the frame is corrupted is permissible in the interrupt, especially if the timing of the operation is essential. For example, if you're receiving CAN frames (writing your own CAN controller instead of using an external one) you have some two bits width worth of time to verify the received frame and assert a resend request; this certainly can't be deferred to the low-priority processing.

As others have suggested, "as little as is possible" is the answer.

There is a school of thought that goes even further, ans suggests that a system should only have ONE interrupt... a timer-tick - with all I/O is performed by polling!

This ensures a known determinacy and provides that the programmer knows exactly what the processor is doing at every cycle - which may be especially relevant in safety-critical applications.

In the case you cite, what you do in the interrupt handler affects whatever else is going on in the system... if the answer is not very much, then validating and processing the data might be OK, but (in general) that is an approach that should not be recommended!
