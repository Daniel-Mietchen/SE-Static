MISRA rule 21.6 and snprintf() (are there any unspecified, undefined, or implementation defined behaviors associated with snprintf?)

up vote 9 down vote favorite

MISRA-C 2012 rule 21.6 is a blanket prohibition of the functions contained within stdio.

The rational "Streams and file I/O have unspecified, undefined and implementation-defined behaviors associated with them."

But snprintf() neither works with streams or with files. It prints to a buffer you provide. Furthermore it takes a size parameter as an argument to mitigate the risk of accidentally writing beyond the bounds of the array pointer you passed in.

I can't figure out what, if any, unspecified, undefined, or implementation defined behaviors could be associated with snprintf.

Are there any? I am strongly considering deviating from 21.6 in the case of snprintf() but want to make sure I am not missing something.

Other answers have established that there is undefined behaviour associated with this function and your position in comments has moved from saying it is safe to just saying you will use it safely.

But, to raise a deviation, MISRA C requires not only that you justify what you are doing is safe, but also that you justify it is necessary.

Since you are just converting plain integers to strings you can implement your own function which safely does only this. Hence there is no necessity for a deviation.

(It should also consume slightly less space and run slightly faster than linking in a full-blown snprintf implementation would have).

There are still multiple ways you can have undefined behavior using snprintf(). For example:

  • lying to it about the size of the buffer you're passing.

    char buffer[100];
snprintf(buffer, 200, "%200s", "BOOM!");

  • providing a different number of format specifiers and arguments.

    char buffer[100];
snprintf(buffer, 100, "%s %d", "BOOM!");

  • mismatch between the format specifier and the type of the argument.

    char buffer[100];
snprintf(buffer, 100, "%200s", 0xB00);

Implementation defined behaviors include:

  • the output format of the %p conversion.
  • the result of converting a double that represents infinity or NaN

From the snprintf manpage:

C99 and POSIX.1-2001 specify that the results are undefined if a call
       to sprintf(), snprintf(), vsprintf(), or vsnprintf() would cause copy‐
       ing to take place between objects that overlap (e.g., if the target
       string array and one of the supplied input arguments refer to the same
       buffer). See NOTES.

< snip >

NOTES
       Some programs imprudently rely on code such as the following

              sprintf(buf, "%s some further text", buf);

       to append text to buf. However, the standards explicitly note that the
       results are undefined if source and destination buffers overlap when
       calling sprintf(), snprintf(), vsprintf(), and vsnprintf(). Depending
       on the version of gcc(1) used, and the compiler options employed, calls
       such as the above will not produce the expected results.

       The glibc implementation of the functions snprintf() and vsnprintf()
       conforms to the C99 standard, that is, behaves as described above,
       since glibc version 2.1. Until glibc 2.0.6 they would return -1 when
       the output was truncated.

(My emphasis)
