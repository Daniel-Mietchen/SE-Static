Redirecting the STDOUT and STDERR streams in IAR Embedded Workbench

up vote 6 down vote favorite

I have my own implementations of PUTC/GETC like functions in my UART driver. I would like to redirect the STDERR and STDOUT streams to one of my UARTs.

In IAR Embedded Workbench for ARM, both of these streams will be directed to the debugger's terminal IO window by default. How can I instead force them to call my UART's PUTC/GETC functions instead.

The goal being, I would like to be able to print debug messages using the standard printf function. But I would also like to be able to perform system level testing without the need to have a debugger connected to the product and still be able to log these messages. I'd also like to have a debug console on a released product that a data logger could be attached to without needing to use the debugger.

I would also like print the resulting message from any asserts that fire to this port, so I will need to hijack stderr as well. I am already #undefining NDEBUG in an assert wrapper.

You need to define functions called __read() and __write() in your application code. They should have the signatures:

__root size_t __write(int handle, const unsigned char *buf, size_t bufSize);
__root size_t  __read(int handle, unsigned char *buf, size_t bufSize);

From there you can call your UART putc()/getc() functions.

The IAR C standard library includes implementations of these functions that send data to the debug console within the Embedded Workbench. They are marked with the __weak attribute so that if you supply those functions, yours will be linked instead.

For more information, see the section "Standard streams for input and output" in the C/C++ Development Guide that comes with the Embedded Workbench.
