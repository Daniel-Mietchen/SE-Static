IAR EWARM 7.4 Placement of a constant with inializers

up vote 4 down vote favorite

I have a structure containing a bunch of configuration parameters. I'd like to hard code the defaults so they are in the binary that is flashed to the product in production. But later on different fields in that section can be updated by the customer. So I will need to be able to put that configuration into a separate flash sector from my code, preferably a small one.

I would like to place it at an absolute location in flash memory with an initializer list.

I am looking at the user guide section: controlling data and function placement in memory and it looks like if I use the #pragma location or the @ operator I have to add __no_init to the declaration. When I do that I can no longer add an initializer list.

So, is there any way to do this? And why can't something called __no_init have initializers if is const - that doesn't add up to me. I feel like that works on other targets with IAR tool chains. It is const so it does not need to be initialized in cstart because it is in flash, but it should still be able to have the initializers because it is the linker that will be placing the values in the image? Am I missunderstanding something?

Two options I'd like to avoid:

  • define another structure separately within const code space and do a memcpy the first time the device powers up.
  • have the linker attach a separate binary to the image at that location
iar-embedded-workbench linker memory-bank

Instead of #pragma location, use #pragma segment:

struct configuration
{
    // Lots of data types...
};

#pragma segment = ".configuration"
const struct configuration user_config @ ".configuration" = 
{
    // Lots of data values...
};

Then override your linker file (the .icf file) to define that segment at the address that you want:

place at address mem:0x12345678 { readonly section .configuration };

By default, the Embedded Workbench uses a .icf file from within its installation directory. I copy that default one into my project directory tree, and edit it by hand (I've never figured out the editor dialog that you can access from the project options).
