What is a clean application layer way of prioritizing scheduling in Linux kernel?

up vote 10 down vote favorite
1

We have a large application that does real time signal processing out of the regular Linux. By large I am implying of the order of 15-18 threads. There could be other regular applications in the same system - but largely this system is used only for the purpose of primarily the main application. All else is lower priority.

We do not yet use RTOS and rely largely on the thread based application on regular Linux kernel. We do not intent to really do kernel programming or switch to RTOS unless it is the only way.

We need some options to be able to instruct the kernel scheduler to maintain thread swtiching in certain rythm. The best thing we know so far is to assign priority but in this case, we have a pipeline where assigning higher priority to anyone thread results in a clogged processing.

My question is, is there a way to assign any thread properties and changing it dynamically through some algorithm such that thread priorities can be varied over time or specify some kind of periodicity for a thread such that all stages of pipeline works in an optimal rhythm?

You can switch the app to SCHED_FIFO priority queue.

This requires a certain way of programming it - essentially, the app needs to work as "cooperative multitasking" one.

#include<sched.h>

struct sched_param param;               
param.sched_priority = sched_get_priority_max(SCHED_FIFO);
if( sched_setscheduler( 0, SCHED_FIFO, &param ) == -1 )
{
        perror("sched_setscheduler");
        return -1;
}

Each thread with SCHED_FIFO scheduler enabled, will run until it yields time to the kernel using sched_yield() or usleep(). [caveat: see below]. Your threads must yield in a regular manner, but they will run uninterrupted until they yield, and other processes will receive any CPU time only when all these threads sleep [or- see below]. That gives an absolute priority to your thread, only being preempted by the kernel, but no userspace threads.

Now for the caveat: /proc/sys/kernel/sched_rt_runtime_us and /proc/sys/kernel/sched_rt_period_us: Your high-priority apps get the exclusive priority for the runtime time of microseconds, and if they don't yield in the meantime, they are preempted for the remainder of the period duration. This is a safety mechanism to prevent total freeze of the system. It can be overridden (set period to 0), at a risk of completely freezing the system, but in reality your app should (by being written that way) yield voluntarily enough time to the system that the mechanism never kicks in - if your app voluntarily sleeps period-runtime microseconds each period, it will never be preempted.

It's a "poor man's realtime" system, but it works if you have one high-priority app. If you need more control, e.g. how often your app wakes up, more fine-grained time slices etc, you need an RT patch for the kernel.

It is possible to set “real-time” priorities in a stock kernel (I tested with 3.18.7 on my Linux PC). You can set a scheduling policy of SCHED_FIFO, SCHED_RR (round-robin), or SCHED_DEADLINE. See sched(7) for details.

To be able to set a priority higher than 0 for a non-root user, RLIMIT_RTPRIO has to be increased. On my Fedora distribution, this is done by giving that user privileges in /etc/security/limits.conf (specifically, setting a hard limit for rtprio). Then that user has to ulimit -r <max priority>. Curiously, you cannot (with ulimit) set the max priority higher than the first value you set it to – you can only decrease it. I don’t quite get what ulimit is doing, since it appears the initial RLIMIT_RTPRIO of a shell is 0. The sched(7) man page says you cannot set the priority higher than the maximum of the current priority and the RLIMIT_RTPRIO soft limit. So you can apparently increase the priority of a shell (and all its child processes) once, but not more than that.

I have been playing around with the examples from here.

As written, they don’t work on my Linux PC which has an 8-core CPU. This is because the processor affinity is set in the main thread, so the second thread never has a chance to get created before the first thread completes (the main thread runs at priority 0). If I move the code to set the affinity to inside the thread, then the examples work as expected, which I can confirm using top. I tweaked the FIFO_1 example to cause thread 1 to delay every so many counts, and this gives thread 2 a chance to run. Thread 2 is otherwise blocked since thread 1 is higher priority.

The PREEMPT_RT patch may not be necessary, depending on the application. It inserts preemption points into the kernel (and makes interrupt handlers “soft interrupts” – they are preemptable kernel threads), so the latency to switch to a thread that becomes runnable is greatly reduced, even when something is running in kernel mode. If the threads you’re concerned about switching don’t spend any time in kernel mode, then the patch doesn’t really buy you much (as long as the maximum time you do spend in kernel mode is acceptably low).

Note that with SCHED_FIFO, the active thread among threads with the same priority will continue to run (before other threads at that priority) until it yields. However, if another thread of a higher priority becomes ready to run, it will do so.
