What is the most reliable and accurate time source available in clock format in Linux x86 systems?

up vote 2 down vote favorite
1

We are using the Intel Core 2 quad core based system in a media server class of application running Linux.

The most critical system is essentially multimedia oriented and hence need to access time in the 'clock ticks' format. The gettimeofday() is available in terms of absolute time which we need to process to convert it as tick. The other alternative clock() function, but it doesn't give consistent result over the long run.

Is there a clock source (or modified version of clock() API) that provide consistent time in terms of 'tick' from the system?

The timer subsystem used by clock() is based on jiffies, which is determined from the CONFIG_HZ kernel config parameter.

From time(7):

The size of a jiffy is determined by the value of the kernel constant HZ.

The value of HZ varies across kernel versions and hardware platforms. On i386 the situation is as follows: on kernels up to and including 2.4.x, HZ was 100, giving a jiffy value of 0.01 seconds; starting with 2.6.0, HZ was raised to 1000, giving a jiffy of 0.001 seconds. Since kernel 2.6.13, the HZ value is a kernel configuration parameter and can be 100, 250 (the default) or 1000, yielding a jiffies value of, respectively, 0.01, 0.004, or 0.001 seconds. Since kernel 2.6.20, a further frequency is available: 300, a number that divides evenly for the common video frame rates (PAL, 25 HZ; NTSC, 30 HZ).

I think you want to make use of the kernel's high resolution timers. You can do this via nanosleep/clock_nanosleep, or with POSIX timers. See timer_create(2) for details.

To see what timers your system supports, you can cat /proc/timer_list.

Also from time(7):

You can determine whether high-resolution timers are supported by checking the resolution returned by a call to clock_getres(2) or looking at the "resolution" entries in /proc/timer_list.

As far as I can tell, there is not a "high-resolution clock" interface to the kernel. I would suggest you get the absolute time at start of your application, and then use high-resolution timers as deltas from the initial clock() call to get a higher resolution of the absolute time.
